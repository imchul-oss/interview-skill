# question_generation.md — BEI/SIT 질문 작성 패턴

**원칙**: 모든 직무에 동일하게 적용되는 질문 생성 알고리즘. 특정 직무의 어휘·시나리오는 박혀있지 않음.

---

## 1. 페어링 원칙 (BEI + SIT)

각 역량에 대해 **BEI 1개 + SIT 1개** 페어링 (필수). 학술 근거:

- **BEI** (*Behavioral Event Interview*, McClelland 1973): 과거 행동 → 미래 예측. "*가장 신뢰할 수 있는 미래 행동 예측 인자는 과거 행동이다*"
- **SIT** (*Situational Interview*, Latham et al. 1980): 가상 시나리오 → 사고·판단 패턴 검증. 경험 부재 후보·신규 영역 탐지에 유효
- **페어링 효과**: 두 방식의 보완으로 cross-validation. 메타분석상 페어링이 단일 방식보다 예측타당도 우수 (*Sackett et al. 2022*).

## 2. BEI 질문 작성 패턴

### 2.1 표준 구조 (한국어)

```
"{역량 핵심 동작}한 경험 중 {수식어}한 사례를 자세히 설명해 주세요. 
{확인하고 싶은 측면}을 함께 알려주시면 좋습니다."
```

**수식어 옵션** (난이도·구체성 유도):
- "가장 도전적이었던"
- "가장 자랑스러운"
- "가장 까다로웠던"
- "가장 의미 있었던"
- "최근 6~12개월 내"

**확인하고 싶은 측면 옵션** (역량별 핵심 검증 포인트):
- "본인의 구체적 역할과 의사결정"
- "그 결과를 정량 지표로 측정한 결과"
- "팀이 한 일과 본인이 한 일의 분리"
- "트레이드오프와 의식적 선택"

### 2.2 BEI 질문 품질 기준

| 기준 | 설명 |
|---|---|
| **개방형** | "예/아니오"로 답할 수 없어야 함 |
| **과거 사례 강제** | 과거 시제 사용, "경험을 설명해 주세요" |
| **단일 사례 유도** | "여러 경험" 대신 "가장 X한 경험 1개" |
| **STAR 자연 유도** | 상황·역할·행동·결과 4요소가 자연스럽게 나오도록 |
| **존댓말** | "~해 주세요", "~알려주시면 좋습니다" |

### 2.3 BEI 부적절 패턴 (금지)

- ❌ "Y/N 질문": "K8s 운영해 본 적 있나요?"
- ❌ "가설 강요": "만약 ~라면 어떻게 하시겠어요?" (이건 SIT 패턴)
- ❌ "역량명 직접 인용": "당신의 리더십을 보여주는 사례는?" (자기 평가 유도, 객관성 약화)
- ❌ "복수 질문": "왜 그렇게 했고, 결과는 어땠고, 다시 한다면 어떻게 하시겠어요?" (꼬리질문으로 분리)

## 3. SIT 질문 작성 패턴

### 3.1 표준 구조

```
"[상황] {구체적 시나리오 — 제약·이해관계자·시간·자원 명시}. 어떻게 {대응/판단/설계}하시겠습니까?"
```

### 3.2 시나리오 구성 요소

| 요소 | 역할 |
|---|---|
| **구체성** | "5억 규모", "1주 데드라인" 등 정량/맥락 명시. 모호함은 답변 깊이 저해 |
| **트레이드오프** | 두 가치 충돌 (속도 vs 품질, 안전성 vs 출시 등). 사고 패턴 검증 |
| **이해관계자** | "고객", "임원", "팀원" 등 대인 차원 포함 |
| **조직 맥락** | 해당 도메인, LLM, 보안 등 산업 맥락 자연스럽게 (강요 아님) |

### 3.3 SIT 질문 품질 기준

- ✅ 후보자가 그 자리에서 사고하도록 강제 (검증된 답이 없는 시나리오)
- ✅ "정답"이 명확히 하나가 아닌 (사고 과정 평가)
- ✅ 후속 꼬리질문이 자연스러운 시나리오

## 4. 직무 중립 시드 (Seed) 패턴

다음은 **모든 직무에 적용 가능한 일반 패턴 카드**. 직무별 시나리오는 JD에서 추출한 키워드로 치환:

### 4.1 BEI 시드 카드

| 카드 | 패턴 |
|---|---|
| **BEI-A** (성과형) | "{역량}을 활용해 {정량 결과}를 만든 가장 자랑스러운 경험을 설명해 주세요." |
| **BEI-B** (난관 극복형) | "{역량} 영역에서 가장 까다로웠던 문제와 해결 과정을 알려주세요." |
| **BEI-C** (협업형) | "{역량}을 발휘하며 다른 팀과 협업한 사례를 설명해 주세요. 의견 충돌이 있었다면 어떻게 풀어가셨는지도 함께 알려주시면 좋습니다." |
| **BEI-D** (학습형) | "{역량}을 빠르게 학습해 실무에 적용하신 경험이 있다면 자세히 알려주세요." |
| **BEI-E** (트레이드오프형) | "{역량} 결정 중 명시적 트레이드오프가 있었던 사례와 그 판단 근거를 설명해 주세요." |

### 4.2 SIT 시드 카드

| 카드 | 패턴 |
|---|---|
| **SIT-A** (위기형) | "[상황] {역량 영역}에서 예상치 못한 문제가 발생했습니다. 어떻게 대응하시겠습니까?" |
| **SIT-B** (신규형) | "[상황] {역량 영역}에서 처음 시도하는 접근이 필요합니다. 어떻게 설계하시겠습니까?" |
| **SIT-C** (충돌형) | "[상황] {이해관계자 A}와 {이해관계자 B}가 {역량 영역}에서 다른 의견을 제시합니다. 어떻게 정렬시키겠습니까?" |
| **SIT-D** (제약형) | "[상황] 시간·예산 제약 속에서 {역량}을 발휘해야 합니다. 어떤 우선순위로 접근하시겠습니까?" |
| **SIT-E** (확장형) | "[상황] {역량} 영역에서 검증된 패턴을 새 도메인으로 확장해야 합니다. 어떻게 진행하시겠습니까?" |

### 4.3 시드 → 직무별 변환

각 역량에 대해 BEI 시드 1개, SIT 시드 1개를 선택하고, JD에서 추출한 **구체 키워드**(도구·시나리오·이해관계자)로 변수 치환.

## 5. 질문 작성 후 자체 검증 체크리스트

INSERT 전:

- [ ] 질문이 완전한 한국어 문장 (면접관이 그대로 읽을 수 있음)
- [ ] BEI는 과거 시제, SIT는 "[상황]" 접두 + "어떻게 ...시겠습니까"
- [ ] 길이: 1~3 문장 (너무 짧으면 모호, 너무 길면 과부하)
- [ ] 특정 회사·인물·제품 이름 박혀있지 않음 (조직은 허용, 사례 회사는 금지)
- [ ] 같은 역량 내 BEI·SIT가 의미 중복 아님

## 6. 기법 카탈로그 — BEI/SIT를 넘어선 다양화 (v0.2.0 신규)

구조화 면접의 핵심은 "같은 직무 = 같은 질문"이다. 그러나 BEI·SIT 두 형식만으로는 일부 역량(특히 실무 수행력·사고 과정)을 충분히 관찰하기 어렵다. 다음 기법을 **표준 질문 세트와 함께 보강 요소로** 설계한다. 각 기법은 표준화된 형태로만 사용하며(후보자마다 동일), 즉흥 비구조화로 흐르지 않는다.

### 6.1 기법별 정의·용도·산출

| 기법 | 무엇 | 언제 | 표준화 방식 | 산출 부착 |
|---|---|---|---|---|
| **BEI** | 과거 행동 사건 | 경험 보유 역량 검증 | 동일 질문 + STAR 꼬리 | question_type='BEI' |
| **SIT** | 가상 상황 판단 | 신규·갭 영역, 사고 패턴 | 동일 시나리오 | question_type='SIT' |
| **관찰 (Observation)** | 답변 중 행동·사고 신호 관찰 | 모든 질문 공통 | 질문별 `observation_cue` 1줄 표준 | 모든 질문에 부착 |
| **워크샘플 (Work Sample)** | 실기·코드 워크스루·화이트보드 | 핵심 역량 갭, 실무 깊이 | 직무별 표준 미니 과제 + 평가 루브릭 | 갭 역량에 면접관 옵션 제안 |
| **케이스 (Case)** | 시나리오 단계 심화 | 시니어 트랙, 설계·판단 | SIT 확장(제약·이해관계자 추가) | SIT 변형으로 표기 |
| **대조 (Contrastive)** | 두 과거 결정·접근 비교 | 판단 근거·메타인지 검증 | 표준 꼬리질문 형태 | BEI 꼬리질문에 추가 |
| **reasoning-aloud** | 사고 과정 소리 내어 풀기 | 문제해결·트레이드오프 | "생각을 말로 풀어 주세요" 표준 지시 | SIT/워크샘플 운용 지시 |

학술 근거: 워크샘플은 *Schmidt & Hunter (1998)* 메타분석에서 예측타당도 r=.54로 구조화 면접(r=.51)과 동급 이상. 관찰·reasoning-aloud는 *Ericsson & Simon (1993) Protocol Analysis* 기반.

### 6.2 관찰 포인트(observation_cue) 작성 규칙

모든 질문(BEI·SIT·MVC)에 답변 중 **무엇을 볼지** 1줄을 부착한다. 점수가 아니라 관찰 안내다.

- 행동 신호: "'우리/나' 구분, 정량 지표 자발성, 트레이드오프 명시 여부"
- 사고 신호: "가설→검증 순서로 사고하는가, 반증을 스스로 탐색하는가"
- 금지: 외모·말투·출신 등 직무 무관 신호 (편향). 직무·가치 관련 행동만.

### 6.3 워크샘플 제안 작성 규칙 (갭 역량 옵션)

매칭 강도 weak/none 인 핵심 역량(가중치 상위)에 한해 면접관 **옵션** 실기 제안을 1줄 생성한다.

- 직무 역량 키워드 기반의 5~10분 미니 과제 (화이트보드·코드 워크스루·설계 스케치).
- 강제 아님, "옵션:" 접두. CV에 없는 사실을 단정하지 않음.
- 평가는 별도 루브릭 없이 BARS 동일 척도로.

### 6.4 기법 선택 결정 규칙 (결정성 보존)

기법 부착은 매칭 강도·질문 유형에 따라 **결정적으로** 정해진다 (LLM 임의 선택 금지).

| 조건 | 부착 기법 |
|---|---|
| 모든 질문 | 관찰 포인트 1줄 |
| question_type=BEI | BEI + 필요 시 대조 꼬리 |
| question_type=SIT | SIT + 시니어 트랙이면 케이스 확장 + reasoning-aloud 지시 |
| 핵심 역량 강도 weak/none | 워크샘플 제안 1줄 (옵션) |
