# structured-interview-q Workflow (5 Steps)

본 워크플로우는 직무 중립적입니다. CV 데이터는 세션 내에서만 사용·폐기됩니다.

---

## Step 1. INITIATE — 발동·입력 받기

**트리거**: `/structured-interview-q` 슬래시 커맨드 또는 발화 패턴 (SKILL.md §2)

### 1.1 첫 안내 메시지
SKILL.md §3 표준 프롬프트 출력.

### 1.2 입력 우선순위 (입력 순서 무관, 모두 갖춰질 때 진행)

| 입력 | 필수 | 미입력 시 |
|---|---|---|
| 직무명 | ✅ | CV에서 추출 시도 → 명확하지 않으면 질문 |
| CV (파일 또는 텍스트) | ✅ | "CV를 입력해 주세요" 요청 |
| 면접 단계 (1차/2차) | ✅ | "1차 면접인가요, 2차 면접인가요?" 질문 |
| 면접 시간 (분) | ✅ | "면접 시간은 몇 분으로 잡으셨나요?" 질문 |
| 트랙 (시니어/미들/주니어) | ⬜ 선택 | 미입력 시 unspecified (CV 추정 금지) |

### 1.2.1 2차 면접 추가 입력 (v0.2.0)

면접 단계가 **2차**이면 입력 시각화(질문지)에 다음 2개를 추가 노출한다 (모두 선택). 입력 결과는 Step 5에서 2차 전용 섹션으로 반영된다 (SKILL.md §5.6).

| 추가 입력 (2차 한정) | 가이드 반영 |
|---|---|
| 추가 검증 희망 영역 | "4.5 추가 검증 포인트" 섹션 |
| 레퍼런스 체크 추가 항목 | "4.6 레퍼런스 체크 포인트" 섹션 |

1차 면접에는 노출·생성하지 않는다. 입력이 비면 해당 섹션을 생략한다 (임의 생성 금지).

### 1.3 직무명 자동 추출 시 검증

CV에서 직무명을 추출했다면:
1. positions 테이블에 매칭되는 직무 검색
2. 매칭 1개 → 사용자에게 확인: "이 후보자는 `{position}` 면접인가요?"
3. 매칭 0개 또는 다중 → 사용자에게 직접 입력 요청

### 1.4 CV 입력 형식

| 형식 | 처리 |
|---|---|
| 텍스트 붙여넣기 | 그대로 사용 |
| .pdf | 텍스트 추출 (pdftotext 또는 PyPDF) |
| .docx | python-docx로 텍스트 추출 |
| .txt | 그대로 |
| .md | 그대로 |
| 기타 | 사용자에게 텍스트 붙여넣기 요청 |

CV 길이 검증: < 200자 → "CV 내용이 충분하지 않습니다" 알림 + 보강 요청.

## Step 2. PARSE & FETCH

### 2.1 CV 파싱 (cv_parsing.md 알고리즘)

CV에서 다음 구조 추출:
```
{
  "candidate_name": "...",      // 식별자만, 민감 정보 제외
  "experience_years": N,
  "current_role": "...",
  "previous_roles": [...],
  "skills": [...],              // 명시된 기술·도구
  "projects": [
    {
      "name": "...",
      "duration": "...",
      "technologies": [...],
      "description": "...",
      "outcome": "..."
    }
  ],
  "education": [...],
  "achievements": [...]         // 정량 성과·수상·논문 등
}
```

**개인정보 가드**: 주소·주민번호·전화번호·이메일 일부 등은 추출하되 출력 .md에는 포함 안 함.

### 2.2 직무 검증 + 표준 질문 fetch

```sql
-- 가드레일: 직무 존재 + 활성 질문 존재
SELECT name, status FROM positions
WHERE name = $1 AND status = 'active';

SELECT * FROM v_active_interview_questions
WHERE position_name = $1
ORDER BY competency_key, question_order;
```

- 직무 미등록 → 거부 + `structured-interview` 안내
- 활성 질문 0행 → 거부 + `structured-interview` 발동 권고
- 정상 → 12 직무 질문 + 15 MVC 질문 = 27 표준 질문 확보

### 2.3 환각 가드 (Critical)

- **interview_questions의 question_text는 절대 수정하지 않음**
- 매칭·인용·우선순위는 출력 .md 생성 시 메타정보로만 처리

## Step 3. MATCH — CV 역량 ↔ 직무 역량 매칭

### 3.1 매칭 알고리즘 (competency_matching.md)

각 직무 역량(C1~C6)에 대해:

| 단계 | 처리 |
|---|---|
| **a. 역량 키워드 추출** | competency_name + JD source 라인에서 핵심 명사구 추출 |
| **b. CV 매칭 검색** | CV의 skills·projects·achievements에서 매칭 검색 |
| **c. 강도 평가** | none / weak / moderate / strong (4단계) |
| **d. 증거 인용** | CV의 어느 부분이 그 강도의 근거인지 명시 |

### 3.2 매칭 결과 분류

| 분류 | 정의 | 면접 전략 |
|---|---|---|
| **🟢 강점 (strong)** | CV에 명확한 경험·정량 성과 | **검증 우선** — 진위·깊이 확인 |
| **🟡 보통 (moderate)** | CV에 일부 경험 | **확장 검증** — 더 깊은 사례 요청 |
| **🟠 약점 (weak)** | CV에 단편적 언급만 | **갭 탐색** — 학습 의지·잠재력 |
| **🔴 갭 (none)** | CV에 전혀 없음 | **잠재력 평가** — 적응력·학습 능력 |

### 3.3 추가 산출

- **Red Flag 후보**: 경력 공백·잦은 이직·모순되는 정보
- **확인 필요 포인트**: 모호한 표현·검증할 주장
- **레퍼런스 체크 포인트** (옵션): 이전 직장에서 검증할 항목

## Step 4. TAILOR — 인용 추가 + 우선순위 산정

### 4.1 BEI 질문에 CV 구체 인용 추가 (question_personalization.md)

**원칙**: 표준 질문 텍스트는 변경 없음. 별도 `cv_specific_addendum` 필드에 인용 추가.

```
[표준 질문]
"React + TypeScript 기반 프로젝트에서 가장 까다로웠던 타입 설계 또는 컴포넌트
 추상화 문제와, 그것을 어떻게 해결했는지 구체적으로 말씀해 주세요."

[CV 기반 추가 (cv_specific_addendum)]
"이력서에 명시된 ''XX 프로젝트''에서의 경험을 중심으로 답변해 주시면 좋습니다."
```

→ 출력 .md에서 표준 질문 + addendum 함께 표시. 면접관이 자연스럽게 합쳐서 진행.

### 4.2 우선순위 산정 (priority_algorithm.md)

**입력**: 면접 단계 + 시간 + 매칭 결과

#### 1차 면접 (직무역량 + 문화 크로스체크)
- **시간 배분**: 직무 70~80% + MVC 20~30%
- **우선순위**:
  1. 핵심 역량 (가중치 ≥ 20%) 강점 검증 (BEI 우선)
  2. 핵심 역량 약점 갭 탐색 (SIT 우선)
  3. 보조 역량 (가중치 < 20%) 시간 허용 시
  4. MVC 5가치 1문항씩 (15분)

#### 2차 면접 (문화 심층 + 리더십)
- **시간 배분**: MVC 60% + 직무 보강 + 리더십 40%
- **우선순위**:
  1. MVC 5가치 BEI+SIT 풀 (40~50분)
  2. 1차에서 미해결 직무 영역 보강
  3. 리더십·성장 가능성 질문

#### 시간 → 질문 수 산정 (참고치)
| 시간 | 권장 질문 수 |
|---:|---:|
| 60분 | 6~8개 |
| 90분 | 10~12개 |
| 120분 | 14~16개 |

(질문당 평균 7~10분, 꼬리질문 포함)

### 4.3 추천 질문 세트 구성

```
[1차 면접 90분 예시]
- 강점 검증 BEI 4개 (핵심 역량 가중치 상위 4)
- 갭 탐색 SIT 2개 (약점·갭 영역)
- MVC 1문항 × 5가치 = 5개
= 총 11개 + 오프닝/클로징
```

## Step 5. ASSEMBLE — 맞춤 가이드 .md 산출

### 5.1 출력 템플릿 (output_personalized.md)

```markdown
# {candidate_name} {job_title} {stage} 면접 가이드

## 후보자 요약
- 경력: {experience_years}년
- 현재 역할: {current_role}
- 핵심 강점: {top_strengths_3}
- 식별된 갭: {identified_gaps}

## 매칭 결과 (역량별)
| 역량 | 가중치 | 매칭 강도 | CV 증거 |
|---|---:|---|---|
...

## 면접 진행 계획 ({stage}, {minutes}분)
| 시간 | 질문 | 우선순위 |
|---:|---|---|
...

## 추천 질문 (우선순위 순)
### 1. {표준 질문}
**+ CV 기반 추가**: {cv_specific_addendum}
- 검증 의도: 강점 확인 / 갭 탐색
- 꼬리질문: {STAR 단계별}

...

## Red Flag / 확인 포인트
- {flag_1} (CV 인용: ...)

## 레퍼런스 체크 포인트 (옵션)
...

## MVC 모듈
{공통 5가치 평가 포인트}

## 참조
- 표준 질문 출처: structured-interview interview_questions (id 인용)
- CV 분석: 세션 내 처리, 영속 저장 안 됨
```

### 5.2 파일명 규칙

```
{candidate_name_safe}_{position_name_safe}_{stage}_{YYYYMMDD}.md
```

예: `김후보_frontend_engineer_1차_20260428.md`

(`_safe` = OS-safe 변환: 괄호·공백 등 제거)

### 5.3 저장 위치

기본: `<workspace>/면접/`. 사용자가 다른 위치 지정 가능.

### 5.4 세션 종료 시 처리

- CV 텍스트 메모리 변수 명시적 폐기
- 출력 .md에는 CV 발췌만 (전문 X)
- 사용자에게 안내: "이 후보자의 CV 데이터는 세션 종료 시 폐기됩니다. 평가 기록을 보존하려면 출력 .md를 안전한 곳에 저장해 주세요."

## 부록: 운영 정책

### A. 표준 질문 변경 감지

스킬 발동 시점에 `interview_questions.sync_status`를 함께 표시:
- `in_sync` → 정상 진행
- `jd_changed_review_needed` → 사용자 알림: "이 직무의 JD가 변경되었습니다. structured-interview에서 먼저 처리해 주세요" → 거부

### B. 다중 후보자 일괄 처리

본 v0.1: 1명씩만. 다중 처리는 v0.2+.

### C. 환각 사고 발생 시

CV에 없는 사실을 인용했음을 발견하면:
1. 즉시 출력 .md 폐기
2. 사용자에게 보고
3. 재실행
